🔄 Data Flow · Token Purchase

Token in Under 10 Seconds

End-to-end journey from user tap to token on screen — happy path

1
User selects DISCO + Meter
Mobile app sends meter number to POST /api/v1/meters/validate/
2
Meter Validation
Backend calls DISCO API. Result cached in Redis (30 min). Meter owner name returned to user for confirmation.
3
User enters amount & pays
App calls POST /api/v1/payments/initiate/. Transaction record created with status PENDING
4
Payment Gateway Callback
Paystack/Flutterwave fires webhook to /api/v1/webhooks/. Signature verified. Transaction → PAID
5
Celery dispatches Token Task
Async worker calls DISCO token API with meter + amount. Retries up to 3× with backoff (2s → 4s → 8s).
6
Token delivered ✓
Token encrypted (Fernet AES-256) and stored. Transaction → COMPLETED. FCM push + SMS sent simultaneously.

🔒 Security Controls
✅  JWT access (15 min) + refresh rotation (30 days)
✅  AES-256 (Fernet) encryption for tokens at rest
✅  TLS 1.2+ on all endpoints (HTTPS only)
✅  Rate limiting per-IP and per-user
✅  Bcrypt password hashing (cost ≥ 12)
✅  Webhook HMAC signature verification
✅  Account lockout after 5 failed attempts
✅  Immutable admin audit log
📡 API Design Principles
✅  RESTful JSON — versioned at /api/v1/
✅  OpenAPI 3.0 (drf-spectacular) at /api/docs/
✅  Stateless — no server-side sessions
✅  Consistent envelope: status · data · message
✅  Pagination (page size 20) on all lists
✅  Structured JSON logs with correlation IDs
✅  ≤ 500ms p95 API response target
✅  Idempotent webhook handlers
